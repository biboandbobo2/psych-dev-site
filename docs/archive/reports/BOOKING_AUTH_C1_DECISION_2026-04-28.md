# Booking Auth-Bypass (C1) — материалы для решения

> **Статус:** open. Ожидается решение от хозяина помещения о приемлемом UX/security trade-off.
> **Источники:** [CODE_REVIEW_MAIN_2026-04-27.md](./CODE_REVIEW_MAIN_2026-04-27.md), [audit-backlog HR-2](../processes/audit-backlog.md#hr-2-закрыть-booking-email-login-auth-bypass).

## Суть уязвимости

Endpoint `POST /api/auth?action=loginByEmail` ([api/auth.ts:20-44](../../api/auth.ts#L20-L44)):

```
1. Body: { email }
2. adminAuth.getUserByEmail(email)
3. Проверка userRecord.emailVerified === true
4. adminAuth.createCustomToken(userRecord.uid)  ← возвращает token
5. Клиент: signInWithCustomToken(auth, token)   ← полноценный логин
```

**Чтобы войти в чужой аккаунт, достаточно знать email.** Никакого второго фактора:

- Нет пароля.
- Нет подтверждения владения почтой **в момент входа**.
- `emailVerified === true` — это property «когда-то подтвердил при регистрации», а не «контролирует ящик прямо сейчас».

CORS wildcard на [api/auth.ts:11](../../api/auth.ts#L11) (`Access-Control-Allow-Origin: *`) позволяет атаку из любого браузера, не только curl. Это часть C2.

### Сценарий атаки

1. Атакующий узнаёт email жертвы (booking — публичная инфа: Telegram-чат центра, рассылки, страница).
2. `curl -X POST https://academydom.com/api/auth?action=loginByEmail -d '{"email":"victim@example.com"}'` → возвращает custom token.
3. `signInWithCustomToken(auth, token)` в любом клиенте → **залогинен как жертва**.
4. Полный доступ: брони, профиль, BYOK Gemini-ключ, заметки, прогресс по тестам, history.

### Почему так получилось

В [src/pages/booking/AuthModal.tsx](../../src/pages/booking/AuthModal.tsx) есть два таба:

- **Регистрация** ([:102](../../src/pages/booking/AuthModal.tsx#L102)) — `sendSignInLinkToEmail` (правильно: магическая ссылка, доказывает владение почтой).
- **Вход** ([:70-87](../../src/pages/booking/AuthModal.tsx#L70-L87)) — `loginByEmail` (быстро, но небезопасно).

Изначально хотели «удобный» одношаговый вход для уже зарегистрированных — без email-cycle. Получилась дыра.

## Варианты решения

### Вариант 1 — Scoped session (custom token claims)

**Идея:** сохранить UX «ввёл email → зашёл», но залогиненная сессия имеет claim `scope='booking'`. Frontend и Firestore rules ограничивают эту сессию только booking-коллекциями.

```ts
// api/auth.ts
const token = await adminAuth.createCustomToken(userRecord.uid, {
  scope: 'booking',
});
```

При следующем логине через email-link новый ID token без custom scope → полный доступ. Scope привязан к **сессии**, не к user record.

**Площадь касания:**

| Слой | Файлы |
|---|---|
| API | `api/auth.ts` (+ claim), все BYOK-endpoint'ы (`assistant`, `lectures`, `books`, `papers`, `transcript-search`) — проверка scope |
| Frontend auth | `src/auth/AuthProvider.tsx` — читать `claims.scope`, прокидывать в context |
| Route guards | `src/app/AppRoutes.tsx` — если `scope='booking'`, везде кроме `/booking/*` редирект |
| Firestore rules | `firestore.rules` — `&& request.auth.token.scope != 'booking'` к каждой коллекции кроме `users/{me}` и booking-related |
| UI | Banner в `/booking` «вы в booking-режиме» + CTA «войти полноценно» |
| Тесты | переписать `booking-login.test.ts`, новые тесты «booking-сессия не читает notes/tests/AI» |
| Docs | `booking-system.md`, `guidelines.md` |

**Объём работ:** ~1-2 дня. Высокий риск «забыли защитить какую-то коллекцию» в rules.

**Остаточные риски** (важно для хозяина):

1. Атакующий по-прежнему может **видеть** будущие визиты жертвы — кто, когда, какой кабинет.
2. **Отменить** чужие брони — DoS / саботаж. Хозяин теряет деньги, жертва приходит к закрытому кабинету.
3. **Создать** бронь от имени жертвы — фейковая бронь на её email.
4. **Обновить** контактные данные жертвы (если booking-flow позволяет редактировать профиль).

Scope защищает только от утечки в **non-booking** области (заметки, тесты, AI, BYOK-ключ).

### Вариант 2 — Booking-OTP без Firebase Auth

**Идея:** booking использует email как identifier без полного Firebase Auth. Каждое посещение — собственный email-link с одноразовым токеном специально для booking.

**Плюсы:** полностью изолирован booking, дыра закрыта на любом уровне.
**Минусы:** нужен свой email-OTP flow (Cloud Function, SendGrid, генерация токенов, validation). Большой объём работы.

### Вариант 3 — Email-link для всех (рекомендую)

**Идея:** любой логин через email-link. Booking использует тот же flow что core-сайт.

**Площадь касания:**

| Слой | Файлы | Объём |
|---|---|---|
| API | `api/auth.ts` — удалить целиком (или урезать) | 1 файл, **бонус: −1 Vercel function (9/12 → 8/12)** |
| Frontend | `AuthModal.tsx:handleEmailLogin` — `sendSignInLinkToEmail` (тот же flow что регистрация, уже есть) | ~10 строк |
| Тесты | `tests/api/booking-login.test.ts` — удалить | 1 файл |
| Docs | `booking-system.md` — обновить flow | 1 файл |
| Firestore rules | **не трогаем** | 0 |
| UI | Объединить tab «Вход» и «Регистрация» в один | минимально |

**Объём работ:** ~30-60 минут.

**Остаточные риски:** ноль — атака физически невозможна без доступа к ящику жертвы.

**Цена для пользователя:** при первом входе — открыть почту, кликнуть. **Firebase запоминает сессию в браузере на 30 дней** — то есть «один раз перешёл — сидит залогиненным месяц». Friction только при первом входе или смене устройства. Это нормальный паттерн (Notion, Linear, Substack).

## Сравнение

| | Вариант 1 (scoped) | Вариант 2 (custom OTP) | Вариант 3 (email-link) |
|---|---|---|---|
| UX «ввёл email → зашёл» | ✅ сохранён | ❌ почта при каждом входе | ⚠️ почта при первом входе (раз в 30 дней) |
| Защита от подделки бронен | ❌ остаётся | ✅ полная | ✅ полная |
| Защита от утечки в non-booking зоны | ✅ | ✅ | ✅ |
| Объём работ | M (1-2 дня) | L (несколько дней) | S (~1 час) |
| Риск ошибок при имплементации | Высокий (Firestore rules) | Средний | Минимальный |
| Освобождение Vercel function | нет | нет | **+1 (9/12 → 8/12)** |

## Что предлагается обсудить с хозяином

Конкретно две вещи:

1. **«Если бывший клиент знает email другого клиента, ему ОК что он может посмотреть и отменить чужую бронь? Или это критично?»**
2. **«Готов ли клиент ради защиты от этого открывать почту и кликать ссылку при первом входе? Или важнее „вошёл сразу“ всегда?»**

Их ответы детерминируют выбор. Это **бизнес-решение**, не техническое.

### Что я бы сказал хозяину дословно

> Сейчас 28 человек могут быть скомпрометированы любым, кто узнает их email. Это не теоретическая дыра — endpoint открыт всему интернету. Из этих 28 — 23 твои арендаторы кабинетов, остальные 5 — администраторы и студенты курса (включая Алексея).
>
> Есть два пути:
>
> 1. **Закрыть полностью.** Арендатор раз в месяц нажимает кнопку «получить ссылку», открывает почту, кликает — и сидит залогиненным 30 дней. Никаких паролей. Это стандарт, как в Notion. Дёшево по работам.
>
> 2. **Оставить как сейчас, но разрешить только booking.** Ввёл email и сразу зашёл, но через этот вход доступен только календарь брони. Чтобы попасть в курс — нужен полноценный вход. Это компрометирует только booking-зону: атакующий с чужим email может видеть, отменять, создавать брони, но не может украсть админ-доступ или BYOK-ключ. Дороже по работе, остаются риски порчи брон.
>
> Что приоритетнее — UX «без открытия почты» или защита от того что чей-то email знакомый/нелояльный сосед посмотрит/отменит чужие брони?

## После решения

- **Если Вариант 3** — можно делать сразу, ~1 час работы. Заодно закрывается C2 (CORS wildcard в `api/auth.ts`) и освобождается одна Vercel function.

- **Вариант 2** — рассматривать только если есть веские причины не использовать Firebase Auth для booking вообще.

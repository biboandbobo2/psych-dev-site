# Timeline: аудит инвариантов (audit/timeline-invariants)

Дата: 2026-07-04. Контекст: после PR #66 (topology tree, B1–B15) каскадные
операции ходят по дереву `buildTimelineTree`/`flattenTree`, но валидаторы в
хуках частично остались на плоском x-матчинге. Цель аудита — явно
сформулировать инварианты, найти классы состояний, которые их нарушают, и
закрыть пробелы тестами.

## Модель данных (wire format, не меняется)

- `NodeT { id, age, x?, parentX?, label, … }` — событие.
  `parentX === undefined | LINE_X_POSITION(2000)` → главная линия;
  иначе `parentX` = `x` ветки, на которой живёт событие.
- `EdgeT { id, x, startAge, endAge, color, nodeId }` — ветка,
  растёт из события `nodeId`, занимает возрастное окно `[startAge, endAge]`.
- Связи закодированы **координатой** (`node.parentX === edge.x`), а не id —
  отсюда весь класс ошибок при совпадении x у разных веток.

## Инварианты

Формат: **зачем** / **где обеспечивается** / **каким тестом** /
**минимальное нарушающее состояние** / **пробел**.

### I1. Каждый node входит в структуру ровно один раз

- Зачем: дубль в `nodes[]` → экспоненциальное размножение при
  drag → flatten → setNodes (реальный инцидент, ~60k событий).
- Где: `buildTimelineTree` — `claimedNodeIds`; `normalizeImportedTimelineData`
  — `dedupeById` на загрузке.
- Тесты: `timelineTree.test.ts` («CRITICAL: … even when two branches share x»,
  «drag flatten/build is idempotent»), `persistence.test.ts`
  («CRITICAL: dedupes nodes/edges by id»).
- Нарушающее состояние: два edge с одинаковым `x` (legacy).
- Пробел: покрыто для build/flatten; **не** покрыто свойство «после любой
  операции (drag, deleteBranch, deleteNode, age-edit) id остаются уникальными».

### I2. Node принадлежит не более чем одной ветви

- Зачем: иначе операции над одной веткой затрагивают чужие события.
- Где: частично — `claimedNodeIds` в `buildTimelineTree` даёт *детерминированную,
  но произвольную* принадлежность: при shared x все события достаются ветке,
  которую обход встретил первой.
- Тесты: только дедупликация (I1); принадлежность не проверяется.
- Нарушающее состояние: `e1(x=2100, from A)`, `e2(x=2100, from B)`,
  событие `n(parentX=2100)`, созданное на `e2` — дерево прикрепит его к `e1`.
- Пробел: **есть**. Drag события A перемещает `n` и переписывает его
  `parentX` на новую координату `e1` — тихая перепривязка чужого события
  (см. Д5). Тестов нет.

### I3. flatten(buildTree(data)) не теряет и не дублирует данные

- Зачем: пара build/flatten выполняется на каждый mousemove при drag;
  любая потеря/дубль немедленно персистится автосейвом.
- Где: `buildTimelineTree` (orphan-узлы поднимаются в корни — данные не
  теряются), `flattenTree`.
- Тесты: `timelineTree.test.ts` («round-trips…», «rescues orphan nodes…»).
- Нарушающее состояние: edge с несуществующим `nodeId` — **сознательно
  дропается** (документировано в модуле); во время drag это тихо перезапишет
  Firestore.
- Пробел: свойство проверено точечно, нет систематической проверки на
  сгенерированном множестве графов (включая пустые ветки, глубокую
  вложенность, дубли x).

### I4. Последовательная нормализация идемпотентна

- Зачем: `normalizeImportedTimelineData` вызывается на каждой загрузке
  документа; не-идемпотентность = «дрейф» данных при каждом открытии.
- Где: `persistence.ts` (dedupe, edge-фильтр, B5-healing, пересчёт ageMax).
- Тесты: идемпотентность **не** проверяется ни одним тестом.
- Нарушающее состояние: кандидатов два — healing меняет `parentX→undefined`
  (повторный проход менять не должен) и `ageMax = max(ageMax, computed)`
  (монотонен, не растёт повторно — надо доказать тестом).
- Пробел: **есть**, закрыть property-тестом `normalize(normalize(d)) ≡ normalize(d)`.

### I5. Удаление ветви не оставляет недостижимых элементов

- Зачем: осиротевшие события копились в Firestore (история B5/B6/B7).
- Где: `applyBranchDeletionToFlat` — миграция событий ветки на родительскую
  линию вместе с внучатыми ветками.
- Тесты: `timelineTree.test.ts` (блок B8, 6 кейсов).
- Нарушающее состояние: (а) сдвиг внучатой ветки `x+deltaX` может попасть на
  `x` существующей чужой ветки → создаёт shared-x (I2/I8 нарушение на
  следующем шаге); (б) мигрированные события получают `parentX` родительской
  ветки, но их `age` может быть вне её `[startAge, endAge]` (нарушение I12).
- Пробел: **есть** — оба следствия не покрыты.

### I6. Удаление события удаляет только его настоящее поддерево

- Зачем: недо-удаление = сироты; пере-удаление = потеря чужих данных.
- Где: `collectDescendantIds` + `deleteNode` (useTimelineCRUD).
- Тесты: `timelineTree.test.ts` (блок B6+B7: root, ветка, лист, сиблинг).
- Нарушающее состояние: shared-x → в поддерево попадают чужие события,
  прикреплённые к «нашей» ветке обходом (I2) → удаляются чужие данные.
- Пробел: **есть** — shared-x кейс для удаления не покрыт.

### I7. Drag не изменяет возраст и не двигает соседние поддеревья

- Зачем: age — семантика данных, x — только презентация.
- Где: `applyDragToTree` меняет только `x`/`parentX`.
- Тесты: B1/B2 блок (сиблинги, внучатые ветки, offset).
- Нарушающее состояние: неизвестно для age (поле не трогается — доказать
  property-тестом «мультидраг сохраняет все age и все id»).
- Пробел: age-сохранность и итеративный drag (много мелких перемещений)
  не проверены свойством; drag может **создать** shared-x (припарковать
  ветку на x чужой ветки) — не покрыто.

### I8. Операции не зависят от совпадения x-координат разных ветвей

- Зачем: x — не ключ; legacy-данные и сами операции (drag, deleteBranch)
  создают совпадения.
- Где: точечно — B15 (`selectedBranchId` вместо x), дедуп в дереве.
  **Остались на x-матчинге:** B11-валидатор (`edges.find(e => e.x ===
  original.parentX)` в useTimelineCRUD), B14-валидатор
  (`n.parentX === selectedEdge.x` в useTimelineBranch), B5-healing
  (`validEdgeXs` по x), членство в дереве (I2).
- Тесты: нет ни одного теста, где два edge с одинаковым x принадлежат
  **разным** родителям.
- Пробел: **системный** — главный кандидат в дефекты (Д4, Д5).

### I9. serialize/deserialize сохраняет семантику данных

- Зачем: экспорт JSON → импорт файла должен быть без потерь.
- Где: `exportTimelineJSON` (прямой JSON.stringify payload),
  `normalizeImportedTimelineData` на входе.
- Тесты: `persistence.test.ts` («normalizes imported timeline data from json
  file») — один happy-path.
- Нарушающее состояние: export → import валидного состояния не должен
  менять nodes/edges (кроме документированного «лечения»).
- Пробел: round-trip свойство не проверено.

### I10. Undo/redo восстанавливает точное предыдущее состояние

- Зачем: единственная страховка от каскадных операций (deleteNode без confirm).
- Где: `useTimelineHistory` + связка в `Timeline.tsx` (`recordHistory`,
  паттерн «пишем в историю состояние ПОСЛЕ действия»).
- Тесты: `useTimelineHistory.test.ts` — только стабильность хендлеров и reset.
  Семантика undo/redo не покрыта вовсе.
- Нарушающие состояния (кандидаты, проверить тестами):
  - **Д1**: baseline после загрузки не записывается → первое действие
    неотменяемо (`canUndo` требует `index > 0`); если первое действие —
    deleteNode (каскадное, без confirm, по хоткею) — данные невосстановимы
    до перезагрузки.
  - **Д2**: `deleteNode` вызывает `onHistoryRecord?.()` без аргументов
    синхронно после `setNodes` → в историю пишется состояние **до** удаления
    (stale closure). Пост-состояние удаления в истории отсутствует → redo
    не может повторить удаление; последовательность undo/redo вокруг
    удаления восстанавливает не те состояния.
- Пробел: **есть**, оба кейса без тестов.

### I11. Импорт повреждённых legacy-данных детерминирован

- Зачем: в Firestore есть документы, испорченные до-B5 багами.
- Где: `normalizeImportedTimelineData`: не-объект → throw; мусорные
  node/edge → отбрасываются; дубли id → первый выигрывает; orphan `parentX`
  → на главную линию; ageMax → пересчёт.
- Тесты: `persistence.test.ts` (B5-блок, dedupe, drop dangling edges).
- Нарушающее состояние: (а) node, чей `parentX` случайно равен x **чужой**
  живой ветки — не лечится, прилипает к чужой ветке (не отличимо от
  легитимного на плоских данных — задокументировать как ограничение);
  (б) события с age вне окна своей ветки импортируются без лечения/отказа
  (I12 нарушен уже на входе).
- Пробел: частичный.

### I12 (производный). Событие на ветке лежит в её возрастном окне

- Зачем: UI рисует событие на отрезке ветки; вне окна — «висит в воздухе»;
  валидаторы B11/B13/B14 существуют ровно для этого.
- Где: B11 (age-edit), B13 (создание ветки), B14 (укорачивание), диапазон в
  BulkEventCreator. **Не** обеспечивается: B10-сдвиг окна ветки при смене
  возраста origin-события (события на ветке не проверяются; `startAge` может
  уйти в минус), миграция при deleteBranch (I5б), импорт (I11б).
- Тесты: B11/B13/B14 — happy + reject; B10-сдвиг с событиями на ветке — нет.
- Пробел: **есть** (Д3).

### I13 (производный). Ссылочная целостность после каждой операции

- `edge.nodeId` указывает на существующий node; `node.parentX` указывает на
  существующий `edge.x` либо главную линию. Обеспечивается на загрузке
  (normalize) и лечится деревом (orphan→root), но ни одна операция не
  проверяется на «не создаёт висячих ссылок» систематически.

## Кандидаты в дефекты (для Этапа 2/3 — сначала падающий тест)

| # | Класс | Суть | Инварианты |
|---|-------|------|-----------|
| Д1 | undo | Baseline не записывается → первое действие неотменяемо | I10 |
| Д2 | undo | `deleteNode` пишет в историю pre-delete состояние (stale closure); redo удаления невозможен | I10 |
| Д3 | age-edit | B10-сдвиг окна ветки не проверяет события на ветке и `startAge ≥ 0`; кламп `endAge` молча укорачивает ветку под событиями | I12 |
| Д4 | shared-x | B11/B14-валидаторы матчат ветку по x → при shared x проверяют чужую ветку | I8 |
| Д5 | shared-x | Членство в дереве при shared x произвольно: drag/delete трогают чужие события и переписывают их `parentX` | I2, I6, I7, I8 |
| Д6 | deleteBranch | Миграция может создать shared-x (сдвиг внучатой ветки) и события вне окна родительской ветки | I5, I8, I12 |
| Д7 | persistence | `shouldPersist` в useTimelineState: очистка последнего непустого холста не сохраняется → после перезагрузки данные «воскресают» | I9 (семантика сохранения) |
| Д8 | import | Идемпотентность normalize не доказана; события вне окна ветки проходят импорт | I4, I11, I12 |

Прим.: Д5 «полное» решение (id-ссылки вместо координатных) меняет wire format
— вне scope. В scope: не ухудшать (операции не должны сами создавать новые
shared-x без необходимости) и зафиксировать текущее поведение тестами.

## Этап 2: тестовая система

Детерминированный генератор малых графов — собственный, без новых зависимостей
(fast-check не нужен: пространство состояний малое, важна воспроизводимость и
читаемые минимальные контрпримеры).

- `src/pages/timeline/utils/__tests__/graphGen.ts` — билдер
  (main line / вложенные ветви / одинаковые x / пустые ветви / глубина ≥ 3 /
  битые parentX / граничные age / повторённые операции) + PRNG (mulberry32,
  фиксированный seed) для перебора небольших случайных графов.
- Новые тест-файлы:
  - `timelineTree.invariants.test.ts` — property-style: I1/I2/I3/I6/I7 на
    множестве сгенерированных графов + regression-кейсы Д5, Д6.
  - `persistence.invariants.test.ts` — I4 (идемпотентность), I9 (round-trip),
    I11 (повреждённые данные), Д8.
  - `useTimelineHistory` + интеграционный тест связки истории — Д1, Д2, I10.

## Журнал результатов

Заполняется по ходу: каждый Дx получает статус
`подтверждён тестом <файл>` / `не подтверждён (ложная гипотеза)` / `зафиксирован как ограничение`.
